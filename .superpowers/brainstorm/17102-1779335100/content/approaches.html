三种优化方案

针对全部 12 个排版问题（含设置页溢出），按改动范围从窄到宽排列

A

最小修正 —— 逐点修补

策略：只修具体的 bug 和明显的硬编码问题，不改变整体布局结构。

  • 修复 overflow: hidden 冲突，设置页独立允许滚动
  • 卡牌尺寸改为相对单位或加 clamp() 缩放
  • AI 容器宽度改为 min-w 而非固定 w
  • 中央区域 gap 和 flex 比例微调
  • 弹窗加 max-w + 响应式尺寸
  • 设置页面板加宽、间距微调

优点

  • 改动文件最少，风险低
  • 快速见效

缺点

  • 仍然是"打补丁"思路，根源问题（缺乏响应式体系）未解决
  • 未来新增页面/组件时同样的问题会重现
B

系统化重构 —— 响应式布局体系（推荐）

策略：建立一套基于 Tailwind 断点的响应式规范，统一解决当前所有问题并为未来打基础。

  • 分离页面级滚动策略：游戏页 overflow-hidden，设置页 overflow-auto
  • 引入 CSS 变量控制卡牌尺寸，通过 clamp() 或 vw 单位实现自适应
  • GameBoard 区域改为 CSS Grid 布局，按比例分配空间
  • AI 手牌区域根据玩家数量动态调整位置和尺寸
  • 玩家手牌加 card-overlap 策略，多牌时自动叠放
  • 设置页面板响应式宽度（max-w-2xl），加内部滚动容器
  • 弹窗统一响应式规范：max-w、padding、按钮尺寸按屏幕适配
  • GameInfo 信息栏加换行适配

优点

  • 根本性解决问题，建立可复用的响应式体系
  • 未来新组件自动遵循相同规范
  • 改动主要在 CSS/Tailwind 层面，不涉及业务逻辑

缺点

  • 改动文件较多（约 8-10 个组件文件 + CSS）
  • 需要更多测试确保不同屏幕尺寸下的表现
C

全面重设计 —— 视觉升级 + 响应式

策略：在方案 B 的基础上，同时升级视觉设计语言。

  • 包含方案 B 的全部内容
  • 重新设计配色方案、阴影层次、圆角规范
  • 弹窗和设置页重新设计 UI 组件（自定义 toggle switch、range slider 等）
  • 加入微交互动效优化
  • 统一设计 Token 系统（间距、颜色、圆角、阴影）

优点

  • 最终效果最好，视觉体验显著提升
  • 建立完整的设计系统

缺点

  • 改动范围大，涉及几乎所有组件
  • 周期长，需要大量视觉验证
  • 风险最高，可能引入新的样式回归

我的推荐

我推荐 方案 B。原因：

  • 12 个问题中大多数根因是"缺乏响应式体系"和"硬编码尺寸"，方案 B 从根本上解决这些结构性问题
  • 改动集中在 CSS/Tailwind 层面，不触碰业务逻辑（游戏引擎、AI、规则等），风险可控
  • 方案 A 只是补丁，同样的问题会在未来重现
  • 方案 C 的视觉升级可以作为独立迭代，不需要和排版修复捆绑
